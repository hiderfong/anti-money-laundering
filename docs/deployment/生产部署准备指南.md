# AML 生产部署准备指南

> 日期：2026-05-10  
> 适用范围：生产或准生产环境部署前检查、部署、验收和回滚。

---

## 1. 部署前目标

生产部署前必须完成以下目标：

- 配置全部通过环境变量或密钥管理系统注入。
- `.env` 中不存在 `CHANGE_ME`、`TODO`、示例域名或开发默认密码。
- `prod` profile 下不暴露 Swagger/Knife4j 文档。
- CORS 只允许生产 HTTPS 域名。
- 数据库迁移、健康检查、核心 E2E、浏览器 E2E 均通过。
- 部署过程可回滚，日志可追踪，备份可恢复。

---

## 2. 准备环境变量

复制模板：

```bash
cp .env.example .env
```

生成密钥：

```bash
openssl rand -base64 48
openssl rand -hex 16
```

填写：

- `AML_JWT_SECRET`：使用 `openssl rand -base64 48` 生成，必须保留 Base64 格式。
- `AML_ENCRYPTION_KEY`：使用 `openssl rand -hex 16` 生成，必须正好 32 个字符。
- `AML_CORS_ORIGINS`：只填写生产 HTTPS 域名，多个域名用英文逗号分隔。
- `AML_API_DOCS_ENABLED=false`：生产默认关闭 API 文档。

如生产环境暂不启用某些外部依赖，可显式关闭：

```env
AML_REDIS_ENABLED=false
AML_KAFKA_ENABLED=false
AML_NEO4J_ENABLED=false
```

Elasticsearch 如不可用，启动时应组合 `no-es` profile：

```env
SPRING_PROFILES_ACTIVE=prod,no-es
```

## 3. 发布前配置检查

运行：

```bash
bash scripts/prod-readiness-check.sh .env
```

检查内容：

- `SPRING_PROFILES_ACTIVE` 是否包含 `prod`。
- 数据库、Redis、Kafka、Elasticsearch、Neo4j、MinIO 配置是否仍为占位值。
- JWT 是否为可解码 Base64。
- 字段加密密钥是否为 32 个字符。
- CORS 是否包含通配符、本机地址或 HTTP 地址。
- API 文档是否在生产环境关闭。

该脚本失败时不要继续部署。

---

## 4. 构建与部署

使用部署脚本：

```bash
bash scripts/deploy.sh <version>
```

示例：

```bash
bash scripts/deploy.sh 20260510-r1
```

脚本会执行：

- 检查 Docker 与 Docker Compose。
- 检查 `.env`。
- 运行 `scripts/prod-readiness-check.sh`。
- 构建 Docker 镜像。
- 停止旧容器。
- 使用 `docker/docker-compose.yml` 启动服务。
- 轮询 `/api/system/health`。

查看日志：

```bash
docker compose --env-file .env -f docker/docker-compose.yml logs -f aml-app
```

停止服务：

```bash
docker compose --env-file .env -f docker/docker-compose.yml down
```

## 5. 部署后验收

服务启动后，先检查健康端点：

```bash
curl -s http://127.0.0.1:8080/api/system/health
curl -s http://127.0.0.1:8080/api/actuator/health
```

再执行 E2E：

```bash
mvn -q test
bash scripts/e2e-test.sh
bash scripts/frontend-e2e.sh
bash scripts/frontend-browser-e2e.sh
bash scripts/rbac-e2e.sh
```

最低限度验收：

```bash
npm --prefix frontend run build
bash scripts/frontend-browser-e2e.sh
bash scripts/rbac-e2e.sh
```

验收重点：

- `/api/auth/me` 返回 `roles`、`permissions`。
- 浏览器 E2E 无新增 `console.error`、`console.warn` 或 `pageerror`。
- RBAC 越权路径返回 `403`，允许路径返回 `200`。
- 健康检查中数据库状态为 `UP`。

## 6. 数据库迁移与备份

项目使用 Flyway：

```text
src/main/resources/db/migration
```

上线前建议：

- 先在准生产环境使用生产数据副本演练迁移。
- 备份 MySQL 数据库。
- 确认迁移脚本只追加，不修改历史 migration。
- 确认 `flyway_schema_history` 正常。

备份示例：

```bash
mysqldump -h "$AML_DB_HOST" -P "$AML_DB_PORT" -u "$AML_DB_USER" -p"$AML_DB_PASSWORD" \
  --single-transaction --routines --triggers "$AML_DB_NAME" > "backup-${AML_DB_NAME}-$(date +%Y%m%d%H%M%S).sql"
```

---

## 7. 回滚策略

建议回滚策略：

- 保留上一版 Docker 镜像 tag。
- 保留部署前数据库备份。
- 如本次发布包含不可逆数据库 migration，发布前必须单独评审回滚方案。
- 应用回滚优先切回上一版镜像；数据库回滚只在确认需要时执行。

应用回滚示例：

```bash
bash scripts/deploy.sh <previous-version>
```
