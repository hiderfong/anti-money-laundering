# AML 业务闭环验收矩阵

> 版本：v2.0  
> 日期：2026-05-10  
> 适用范围：当前本地集成版，后端 `8080`，前端 `5173`  
> 基线证据：API E2E `18/18` 通过，前端 E2E `31/31` 通过，浏览器实测 `admin/<local-test-password>` 登录、仪表盘、系统管理页可访问

---

## 1. 验收目标

当前项目已经具备可运行的前后端集成基础，但现有脚本更偏向“接口可用、路由可达”。本矩阵用于把后续验收升级为“真实业务闭环可证明”，覆盖：

- 用户登录、Token 刷新、RBAC 权限一致性。
- 客户建档、风险评估、名单筛查、交易录入、预警生成、预警处理、案件调查、STR/大额报告报送。
- 页面操作、接口响应、数据库状态、审计日志、通知提醒之间的一致性。
- 测试数据可识别、可重复运行、可清理。

---

## 2. 通用验收标准

| 类别 | 验收标准 |
|------|----------|
| 功能正确性 | 页面操作结果、API 返回值、数据库状态三者一致 |
| 权限一致性 | 菜单、路由、按钮、后端 API 同时符合角色权限矩阵 |
| 状态流转 | 预警、案件、报告等状态只能按允许路径流转 |
| 可追溯性 | 新增、修改、审核、关闭、报送等关键操作写入审计日志 |
| 测试隔离 | E2E 数据使用统一前缀，测试结束可自动清理 |
| 可重复性 | 同一脚本连续运行不依赖上一次残留数据 |
| 前端体验 | 有加载态、空态、错误态、表单校验、提交反馈 |
| 失败诊断 | 脚本失败时输出当前步骤、请求 URL、响应摘要、截图或 DOM 摘要 |

## 3. 测试数据规范

| 数据类型 | 命名规则 | 示例 |
|----------|----------|------|
| 客户姓名 | `E2E客户_${timestamp}` | `E2E客户_20260510153000` |
| 交易编号 | `E2E_TXN_${timestamp}_${seq}` | `E2E_TXN_20260510153000_001` |
| 案件编号 | 系统生成，备注写入 `E2E_CASE_${timestamp}` | `E2E_CASE_20260510153000` |
| 报告备注 | `E2E_REPORT_${timestamp}` | `E2E_REPORT_20260510153000` |
| 用户名 | `e2e_${role}_${timestamp}` | `e2e_viewer_20260510153000` |

清理策略：

- 优先通过业务删除/关闭接口清理。
- 无删除接口时，按 E2E 前缀记录待清理 ID，使用测试专用清理脚本处理。
- 清理脚本默认只处理 E2E 前缀数据，禁止无条件清空业务表。

## 4. 业务闭环验收矩阵

### AML-E2E-01 登录、Token 与用户画像

| 项目 | 内容 |
|------|------|
| 目标 | 验证登录、Token 刷新、当前用户画像、前端会话恢复 |
| 角色 | `ROLE_ADMIN` |
| 页面 | `/login`、`/dashboard` |
| API | `POST /auth/login`、`POST /auth/refresh`、`GET /auth/me` |
| 前置数据 | 管理员账号 `admin/<local-test-password>` |
| 操作步骤 | 登录；读取本地 Token；刷新 Token；刷新页面；访问 `/dashboard` |
| 期望结果 | 返回 `accessToken`、`refreshToken`、`roles`、`permissions`；刷新后仍保持登录态 |
| 证据 | API 响应包含 `ROLE_ADMIN`；浏览器 URL 为 `/dashboard`；无控制台错误 |
| 自动化优先级 | P0 |

### AML-E2E-02 RBAC 四层一致性

| 项目 | 内容 |
|------|------|
| 目标 | 验证角色权限在菜单、路由、按钮、API 四层一致 |
| 角色 | `ROLE_ADMIN`、`ROLE_COMPLIANCE`、`ROLE_INVESTIGATOR`、`ROLE_VIEWER` |
| 页面 | `/kyc`、`/screening`、`/monitoring`、`/alerts`、`/cases`、`/reporting`、`/system` |
| API | `/auth/me`、各模块核心读写接口 |
| 前置数据 | 为每个角色准备独立 E2E 用户 |
| 操作步骤 | 登录各角色；检查菜单；直接访问受限路由；调用受限 API；检查按钮显示 |
| 期望结果 | 无权限菜单不显示；无权限路由进入 `/403`；无权限 API 返回 403；按钮权限符合 `permissions` |
| 证据 | 角色权限矩阵报告、路由截图、API 状态码 |
| 自动化优先级 | P0 |

## 6. 当前已知差距

| 差距 | 影响 | 建议 |
|------|------|------|
| 现有脚本主要验证 HTTP 200 和核心 API 可用 | 不能证明真实页面操作闭环 | 增加 Playwright 或等价浏览器 E2E |
| E2E 会产生客户、交易等测试数据 | 长期运行会污染本地库 | 增加 E2E 前缀和清理脚本 |
| 多角色账号准备不足 | RBAC 只能验证管理员路径 | 增加角色种子数据或测试用户创建流程 |
| 状态机验收未形成硬约束 | 预警、案件、报告可能出现非法跳转 | 为状态流转补单元测试、集成测试和 UI 断言 |
| 前端 `feedbackApi` 路径与后端规则反馈 Controller 路径不一致 | 对应页面或后续调用可能 404 | 在实施计划中列为接口契约修复项 |
