### 1.2 AML Core Engine — Java + Spring Boot AML 监控服务（直接参考）

| 项目 | 信息 |
|------|------|
| GitHub | https://github.com/ByteEntropyDotCom/aml-core |
| 语言 | Java 21 + Spring Boot 3.4 |
| 依赖 | Redis, Maven |

**核心特性：**
- 基于 Java 21 Virtual Threads（Project Loom）的高并发处理
- 全异步管道（CompletableFuture + @Async），API 响应 < 5ms
- Redis Lua 脚本实现原子性速度评估（防竞态条件）
- 业务规则与代码解耦，通过配置文件驱动

**技术栈：**

| 技术 | 用途 |
|------|------|
| Java 21 | Virtual Threads 大规模并发 |
| Spring Async | 非阻塞交易处理管道 |
| Redis | 低延迟状态跟踪（用户交易频率） |
| JUnit 5 | 集成和单元测试 |
| Maven | 构建管理 |

**核心规则示例：**

```properties
# 速度规则：120秒内超过10笔转账 → 触发预警
aml.rules.velocity.threshold=10
aml.rules.velocity.window-seconds=120

# 大额规则：单笔超过10,000美元 → 触发预警
aml.rules.high-value.limit=10000.00

# 线程池配置
spring.task.execution.pool.core-size=8
spring.task.execution.pool.max-size=50
spring.task.execution.pool.queue-capacity=10000
```

**对我们项目的借鉴价值：**
- 技术栈高度匹配（Java + Spring Boot + Redis）
- Virtual Threads 方案可直接用于高并发交易监控
- Redis Lua 脚本做原子性规则评估的实现方式
- 规则配置外部化的设计模式
- 异步处理管道的架构设计

### 2.2 保险行业 AML 实践案例

#### 案例一：某大型寿险公司 AML 系统建设

**背景：** 人寿保险产品（尤其是万能险、投连险）具有较高洗钱风险

**系统架构：**
```
保全/理赔系统 ──→ ETL ──→ AML 数据仓库
                              ↓
                    ┌─────────────────┐
                    │  规则引擎层       │
                    │  - 大额交易规则   │
                    │  - 频率规则       │
                    │  - 退保异常规则   │
                    │  - 保单贷款规则   │
                    └─────────────────┘
                              ↓
                    ┌─────────────────┐
                    │  ML 模型层       │
                    │  - 异常检测       │
                    │  - 客户聚类       │
                    └─────────────────┘
                              ↓
                    ┌─────────────────┐
                    │  预警管理         │
                    │  - 人工审核       │
                    │  - 案件分配       │
                    │  - 调查处理       │
                    │  - 报告生成       │
                    └─────────────────┘
                              ↓
                    人民银行反洗钱监测分析系统
```

**关键指标：**
- 日均处理保单交易：50万+
- 规则命中率：约 2%-5%
- 人工审核后确认可疑：约 0.1%-0.5%
- 平均案件处理时间：2-3 个工作日

#### 案例二：保险业制裁名单筛查实践

**筛查节点：**
1. 新保单投保时
2. 保单信息变更时
3. 理赔支付时
4. 退保支付时
5. 保单贷款发放时
6. 存量客户定期复查（至少每季度）

**匹配算法：**
- 精确匹配（身份证号、护照号）
- 模糊匹配（姓名相似度，支持中英文、多音字）
- 组合匹配（姓名 + 出生日期 + 国籍）
- 阈值可配置（推荐相似度 >= 85% 进入人工复核）

**误报处理：**
- 建立白名单机制
- 记录排除原因
- 定期回顾排除案例

### 3.2 规则引擎选型对比

| 方案 | 优势 | 劣势 | 适用场景 |
|------|------|------|---------|
| **Drools** | 成熟、社区活跃、支持复杂规则 | 学习曲线较陡、内存占用大 | 规则复杂度高、需要热更新 |
| **Easy Rules** | 轻量、易上手、Java 原生 | 功能有限、不适合复杂规则 | 规则简单、快速启动 |
| **自研规则引擎** | 完全可控、性能优化空间大 | 开发成本高、维护负担重 | 有特殊定制需求 |
| **Redis + Lua** | 高性能、原子操作、分布式友好 | 规则表达能力有限 | 速度/频率类规则 |
| **Apache Flink** | 流式处理、复杂事件处理(CEP) | 架构复杂、运维成本高 | 实时流式监控 |

**推荐方案：混合使用**
- Drools：复杂业务规则（可疑交易识别逻辑）
- Redis + Lua：速度/频率类规则（短时间内高频交易）
- 自研轻量引擎：大额阈值等简单规则

### 3.3 名单匹配技术方案

| 算法 | 适用场景 | 精度 | 性能 |
|------|---------|------|------|
| 精确匹配 | 证件号码 | 最高 | 最快 |
| Jaro-Winkler | 英文姓名相似度 | 高 | 快 |
| Levenshtein | 通用字符串编辑距离 | 中 | 中 |
| 编辑距离 + 拼音 | 中文姓名匹配 | 高 | 中 |
| TF-IDF + Cosine | 长文本相似度 | 中 | 慢 |
| Elasticsearch | 全文检索 + 模糊匹配 | 高 | 快（索引后） |
| 向量嵌入 + ANN | 语义相似度 | 高 | 快（索引后） |

**推荐方案：**
- 英文名：Jaro-Winkler + 阈值
- 中文名：拼音转换 + 编辑距离 + 多音字处理
- 证件号：精确匹配
- 高性能场景：Elasticsearch 或 Redis 模块

### 3.4 数据存储方案

| 数据类型 | 推荐存储 | 说明 |
|---------|---------|------|
| 客户信息 | MySQL / PostgreSQL | 结构化、事务保障 |
| 交易流水 | MySQL 分库分表 + Elasticsearch | 大量写入 + 全文检索 |
| 名单数据 | Redis + Elasticsearch | 内存加速 + 模糊匹配 |
| 规则配置 | MySQL + Redis 缓存 | 持久化 + 热加载 |
| 预警案件 | MySQL | 结构化、流程管理 |
| 审计日志 | Elasticsearch | 全文检索、长期保留 |
| 报表数据 | ClickHouse / 列式存储 | 聚合分析性能 |

## 四、关键技术难点与解决方案

### 4.1 中文姓名模糊匹配

**难点：** 多音字、简繁体、同音字、少数民族姓名

**解决方案：**
```
输入姓名
    ↓
拼音转换（支持多音字词典）
    ↓
相似度计算（编辑距离 + Jaro-Winkler 加权）
    ↓
上下文校验（出生日期、证件号辅助确认）
    ↓
阈值判断（>= 85% → 人工复核，>= 95% → 自动命中）
```

### 4.2 实时交易监控性能

**难点：** 保险交易量大、规则复杂、要求低延迟

**解决方案：**
- 采用 Java 21 Virtual Threads 处理高并发
- Redis Lua 脚本做原子性速度计算
- 异步管道解耦接收和分析
- 规则分层：简单规则内存计算，复杂规则异步处理
- 批量交易预聚合（滑动窗口）

### 4.3 可疑交易识别准确率

**难点：** 误报率高、规则调优困难

**解决方案：**
- 规则 + ML 双重验证降低误报
- 基于历史案件反馈持续优化规则阈值
- 引入客户画像和行为基线
- 多维度评分加权而非单一规则触发
- 建立误报反馈闭环

### 4.4 监管报送数据格式

**难点：** 人民银行接口规范复杂、XML 格式严格

**解决方案：**
- 抽象报送数据模型
- 模板化 XML 生成
- 数据校验层（格式、完整性、逻辑校验）
- 报送前预览和人工确认
- 失败重试和状态追踪

## 五、项目实施建议

### 5.1 分阶段建设路线

```
第一阶段（MVP，3-4个月）
├── 基础客户信息管理（KYC）
├── 大额交易监测与报告
├── 基础制裁名单筛查（精确匹配）
├── 简单案件管理工作流
└── 监管报送基础功能

第二阶段（增强，2-3个月）
├── 可疑交易规则引擎
├── 客户风险等级划分
├── 模糊名单匹配
├── 受益所有人识别
└── 报表与仪表盘

第三阶段（智能化，2-3个月）
├── ML 异常检测模型
├── 交易网络关联分析
├── 规则自学习与调优
├── 高级报表与分析
└── 与核心系统深度集成
```
