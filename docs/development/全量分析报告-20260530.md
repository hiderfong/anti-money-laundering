## 三、当前主要问题（按优先级）

### P1 — 应优先处理

| 编号 | 问题 | 证据 | 影响 |
|------|------|------|------|
| **P1-A** | **审计日志等敏感端点缺少授权（RBAC 粒度缺失）** | `AuditLogController`（`/system/audit-logs` 含 `/export`）、`NotificationController`、`RuleFeedbackController`、`GraphAnalysisController` 均 **0 个 `@PreAuthorize`**。安全基线是 `anyRequest().authenticated()`，故这些端点**任何已登录用户（任意角色）均可访问** | 审计日志可被任意用户读取并导出；对合规系统，审计日志通常应限 admin/审计/合规角色。这是最尖锐的一项 |
| **P1-B** | **监管报送链路（STR/CTR）零单元测试** | `reporting` 模块 4 个 service 接口、`StrReportController` 等无单测；`assessment`、`prevention`（含 454 行 `SpecialPreventionServiceImpl`）、`rectification` 同样 0 测试 | 反洗钱系统最受监管关注的"可疑/大额交易报送"与"自评估/特殊防控"逻辑缺回归保护 |
| **P1-C** | **前端零测试** | 51 个 Vue/TS 文件、19,499 行，无任何单元/组件测试；仅有 `frontend-browser-e2e.mjs` 一个 Playwright 脚本 | 前端逻辑（权限路由、表单校验、i18n）回归全靠手测 |

## 四、改进建议（按优先级）

1. **立即（P1-A）**：为 `AuditLogController`、`NotificationController`、`RuleFeedbackController`、`GraphAnalysisController` 补 `@PreAuthorize`。审计日志读取/导出至少限 `hasRole('ADMIN') or hasAuthority('audit:view')`；其余按既有 RBAC 矩阵（见 `docs/development/RBAC角色权限矩阵.md`）对齐。改动小、风险低、可立即做。
2. **近期（P1-B）**：为监管报送（STR/CTR）与特殊防控补关键路径单测/集成测试——这是合规系统回归保护的优先级最高处。
3. **近期（P2-A）**：默认 `application.yml` 移除 `env` 暴露；`SecurityConfig` 的 `PERMIT_ALL_PATHS` 仅放行 `/actuator/health`，其余 actuator 端点要求认证。
4. **规划（P1-C）**：引入前端组件测试（Vitest + Vue Test Utils），优先覆盖权限路由守卫与表单校验。
5. **规划（P2-C/P2-D）**：按 `AiRiskScoringServiceImpl` 拆分范式处理三大单体服务；为训练拉取补 recency-window cap。
6. **清理（P2-B）**：删除 3 个未路由死视图，确认无残余引用。
7. **文档（P3）**：建现行状态索引页。
