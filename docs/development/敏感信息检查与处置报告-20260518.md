## 3. Git 历史判断

历史中仍可看到旧的本地开发默认值相关提交，这是开发样例与测试数据历史，不包含本次用户提供的远程服务器口令、远程管理员口令或 GitHub token。

当前仓库保持私有时，暂不建议立即重写历史，因为重写历史会影响已有 clone、tag、release、Actions 基线和远程部署对齐。若后续计划公开仓库或执行严格合规清理，应单独安排一次历史重写窗口，使用 `git filter-repo` 清理旧开发默认值，并强制推送后通知所有协作者重新克隆。

## 4. 远程部署发现

远程服务器当前仍存在运行配置风险：

- 后端 Java 进程仍以 `dev,no-es` profile 运行。
- 开发中间件容器端口曾以公网监听方式运行，包括数据库、Redis、Neo4j、MinIO 等。
- 远程仓库工作区版本落后于本地当前主线。

建议下一步在维护窗口内完成远程整改：

1. 备份远程数据库与当前部署目录。
2. 将远程中间件端口绑定收紧为 `127.0.0.1`。
3. 生成远程生产 `.env`，通过 `prod,no-es` 或等价生产 profile 启动后端。
4. 轮换数据库、Neo4j、MinIO、JWT、字段加密等密钥。
5. 运行 `scripts/prod-readiness-check.sh`、健康检查、登录与核心业务烟测。
