## 结论

当前发布候选的代码质量门、生产配置门、前端/后端基础构建、GitHub Actions 和本机 Docker 镜像构建均已通过。GitHub Actions 已由人工确认全绿，且 Node.js 20 action deprecation warning 已消失。

本机发布演练已进一步完成最小依赖部署验证：在 MySQL 与 MinIO 启动、Redis 与 Elasticsearch 明确关闭的场景下，应用容器启动成功，`/api/system/health` 与 `/api/actuator/health` 均返回健康状态。验证过程中发现 Docker Compose 需要透传 `SPRING_AUTOCONFIGURE_EXCLUDE`，否则关闭 Redis 后 Redisson 自动配置仍会尝试连接 `redis` 主机；本轮已将该环境变量加入 Compose 配置，并补充到生产部署指南。

完整依赖栈的本机演练仍受本机环境限制：Kafka 镜像拉取遇到镜像代理 `429`，Elasticsearch 在当前 Colima 资源下以 `exit code 137` 退出。该问题更接近本机 Docker/资源环境限制，不构成当前应用镜像或发布脚本失败；完整依赖栈仍建议在准生产环境或资源更充足的外部 Agent 上复验。

## 需要关注的环境差异

| 项 | 现象 | 判断 |
| --- | --- | --- |
| Maven 默认 JDK | `mvn -version` 显示 Java 25 | 需显式设置 JDK 21；项目 Enforcer 正确阻止非 JDK 21 构建 |
| Maven 默认仓库 | 沙箱不能写 `~/.m2` tracking 文件 | 已用 `/tmp/aml-m2-release` 隔离仓库绕过，不影响代码判断 |
| Docker CLI | 初始缺少 `buildx`，后通过 Homebrew 与 Docker CLI 配置修复 | 本机已修复；其他机器需确认 `docker buildx version` |
| Docker Compose | `docker compose` 不可用，`docker-compose 5.1.1` 可用 | 当前脚本可自动走 legacy `docker-compose` 路径 |
| Kafka 镜像拉取 | `bitnami/kafka:3.6` 拉取遇到镜像代理 `429` | 本机网络/镜像源问题；准生产应使用稳定镜像仓库或预拉取镜像 |
| Elasticsearch 本机启动 | 当前 Colima 环境中 ES 以 `exit code 137` 退出 | 本机资源限制概率高；完整依赖栈建议在资源充足环境复验 |
