## 5. 启动命令记录

### 5.1 基础设施
```bash
# MySQL + Redis
docker compose -f docker/docker-compose-dev.yml up -d mysql redis

# Kafka
docker run -d --name aml-kafka --network host \
  -e KAFKA_NODE_ID=1 \
  -e KAFKA_PROCESS_ROLES=broker,controller \
  -e KAFKA_CONTROLLER_QUORUM_VOTERS=1@127.0.0.1:9093 \
  -e KAFKA_LISTENERS=PLAINTEXT://0.0.0.0:9092,CONTROLLER://0.0.0.0:9093 \
  -e KAFKA_ADVERTISED_LISTENERS=PLAINTEXT://127.0.0.1:9092 \
  -e CLUSTER_ID=MkU3OEVBNTcwNTJENDM2Qk \
  apache/kafka:3.7.0

# Elasticsearch
docker run -d --name aml-elasticsearch \
  -p 127.0.0.1:9200:9200 \
  -e discovery.type=single-node \
  -e xpack.security.enabled=false \
  -e "ES_JAVA_OPTS=-Xms256m -Xmx256m" \
  elasticsearch:8.12.0
```

## 8. Codex 复核意见

> 复核时间：2026-05-13

该报告可以作为 `v0.1.0-rc.1` 的完整依赖环境集成演练证据：MySQL、Redis、Kafka、Elasticsearch、MinIO 均已启动，后端单元测试、API E2E、RBAC E2E 全部通过，Kafka 事件写入与 Elasticsearch 集群健康均有明确记录。

但报告中的后端启动命令使用了：

```bash
-Dspring-boot.run.profiles=dev,no-redis
```

`no-redis` profile 会关闭 Redis/Redisson 自动配置，因此本报告不能单独证明生产形态下应用容器通过 Redis 密码认证连接 Redis。该点不否定本次演练结论，但应将其定性为“完整依赖环境 + 业务链路集成演练通过”，而不是“生产 Compose 全栈部署演练完全通过”。

发布前仍建议在准生产或资源充足 Docker 环境中补跑一次生产形态验证：

```bash
SPRING_PROFILES_ACTIVE=prod
AML_REDIS_ENABLED=true
bash scripts/prod-readiness-check.sh .env
bash scripts/deploy-preflight.sh .env
bash scripts/deploy.sh v0.1.0-rc.1
```

验收重点：

- Redis 使用生产密码模式，应用日志无 Redisson/Redis 连接失败。
- Kafka 使用目标镜像源或预拉取镜像，不再依赖不稳定镜像代理。
- Elasticsearch 在目标 Docker 资源限制下稳定运行，健康状态为 `green` 或可接受的 `yellow`。
- `/api/system/health`、API E2E、RBAC E2E、前端浏览器 E2E 全部通过。
