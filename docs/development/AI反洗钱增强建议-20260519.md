# AI 反洗钱能力增强建议

> 生成日期：2026-05-19
> 代码基线：`6a2d675` (main)
> 范围：聚焦 AI/ML 反洗钱识别能力，附带工程侧补充项

---

## 一、当前 AI/ML 能力的客观评估

| 组件 | 实质 | 关键事实（基于代码） |
|------|------|----------------------|
| `AiRiskScoringService`（"AI 可解释风险评分基线 v1"） | **规则化加权打分，非机器学习** | 分数 = 手工设定的整型因子贡献求和后 clamp(0,100)；类注释明确写"v1先以规则化特征评分方式落地…后续可接监督学习模型" |
| `TransactionAnomalyDetector` | Smile Isolation Forest（无监督） | 仅 `@PostConstruct` 启动时训练一次（窗口 `aml.ml.training-days:90`，阈值 0.7），模型序列化到磁盘；**无 `@Scheduled` 周期重训**，无漂移监控 |
| 人工复核反馈（V015：`manual_review_label/comment/reviewed_by`） | **只采集，不回流** | `module/ai` 内无任何 retrain/feedback 接线；标注数据在持续积累却未被任何模型使用 |
| `RuleFeedbackService` | 规则命中率/误报率统计 | 纯分析展示，无自动阈值调优、无模型 |
| Neo4j 图分析 | 环路/共享账户/多层转移/网络密度查询 | `@ConditionalOnProperty` 可选；**结果未作为特征进入 AI 评分**，评分器只用简单 SQL 计数 |

**核心结论**：系统已具备"可解释规则评分 + 无监督异常检测 + 图查询"三块基础，但三者**彼此孤立**，且**最有价值的资产——人工复核标注——完全未被利用**。当前"AI"主要是营销命名，真实 ML 含量低。

---

## 二、AI/ML 增强路线（按投入产出排序）

### P0｜闭环监督学习：让复核标注回流（最高 ROI）

现状：`t_ai_risk_score_record` 已沉淀 `manual_review_label`（TRUE_POSITIVE/FALSE_POSITIVE/NEEDS_MONITORING）+ 完整 `feature_snapshot_json`。这正是一份**带标签的训练集**，却被丢弃。

建议：
1. 新增 `AiRiskModelTrainingService`：定期（如每周 `@Scheduled` 或 modelmgmt 触发）拉取已复核记录，以 `feature_snapshot_json` 为特征、`manual_review_label` 为标签，训练**监督模型**（梯度提升树 XGBoost/LightGBM via JPMML，或先用逻辑回归基线）。
2. 产出**校准后的概率**（Platt scaling/Isotonic），替代当前不可比的整型求和分。
3. 通过 `AmlModel`（modelmgmt 已有生命周期字段）做**冠军/挑战者**：新模型先影子运行，指标达标再 `DEPLOYED`。`AiRiskScoringServiceImpl.resolveServingModel()` 已是天然接入点。
4. 保留现规则分作为**可解释回退**与冷启动，模型分与规则分并行展示。

价值：把"伪 AI"变成真正自学习系统；复核越多模型越准，形成数据飞轮。

### P1｜异常检测产品化

- `TransactionAnomalyDetector` 增加 `@Scheduled` 周期重训 + 训练成功率/样本量告警；当前只在启动时训练，长期运行模型必然陈旧。
- 引入**模型漂移监控**：特征分布 PSI、异常分分布偏移，触发重训或预警（可复用 `RuleFeedbackService` 的统计风格）。
- Isolation Forest 异常分目前只作为评分器一个 +12/+18 的因子；建议将其**校准为概率**并纳入集成模型特征。

### P2｜图特征融入评分（结构性洗钱信号）

洗钱的核心是资金网络结构，但 `AiRiskFactorEvaluator` 仅用 `distinctCounterpartyCount`、`sharedCounterpartyAccountCustomerCount` 等浅层 SQL 计数。

建议把 Neo4j 已有能力**特征化**后喂给模型/评分：
- 网络中心性（PageRank/度中心性）、社区检测（Louvain）所属社区风险浓度、是否处于资金环路、最短路径触达制裁实体的跳数、过账链路深度。
- 中长期可上图神经网络（GNN）做关系级可疑识别（GraphSAGE/GCN），但需先把上面的图特征工程做扎实。

### P3｜实体解析与名单智能匹配

- 现 `NameMatcher` 为字符串/拼音匹配。建议增强：模糊匹配 + 别名/音译归一 + 嵌入向量相似度（多语种），降低制裁名单漏报与误报。
- 客户实体消歧（同人多账户/多证件聚类），为图特征与 KYC 提供更准的实体粒度。

### P4｜可解释性与合规对齐（监管要求，不可省）

- 监督模型上线后必须配 **SHAP/特征贡献**输出，复用现有 `AiRiskFactorVO`（factorCode/contribution/evidence/suggestion）结构承载，保持"每个分数可向监管解释"。
- 记录模型版本、训练数据范围、特征快照（已具备 `feature_snapshot_json`）形成**可审计模型档案**，对接 `module/modelmgmt`。
- 增加偏差/公平性检查（避免对特定职业/地域系统性高估）。

### P5｜场景扩展

- 典型洗钱模式专项模型：结构化拆分（smurfing）、快进快出（pass-through）、保单退保套现、可疑受益人变更。
- 序列模型（交易时序）识别"养卡—激活—集中转出"等时间模式，当前评分是快照式、无时序建模。

## 三、工程侧补充增强（非 AI，但支撑 AI 可靠落地）

- **特征存储/一致性**：训练用 `feature_snapshot_json` 与在线 `AiRiskFeatureBuilder` 须保证同源同口径，否则训练-服务偏斜（training-serving skew）。建议特征构建逻辑单一出口（已拆出 `AiRiskFeatureBuilder`，是好起点）。
- **回测框架**：模型/规则变更前用历史已结案数据回测召回/精确率，纳入 CI 发布门禁。
- **`AiRiskFactorEvaluator` 因子权重外置**：当前 +32/+18 等硬编码在代码里，建议迁到 `module/modelmgmt` 可配置，便于合规调参与审计（无需发版）。
- 性能：`AiRiskFeatureBuilder` 每次评分发多条聚合 SQL，批量评分场景需缓存/物化，否则无法支撑全量客户定期重评分。
- 承接前序报告未决项：**P1-3**（notification 数据契约根治）、**P2-4**（中间件栈过重）仍需业务/架构决策。

## 四、建议优先级总览

| 优先级 | 事项 | 依赖 | 预期收益 |
|--------|------|------|----------|
| **P0** | 复核标注回流 → 监督模型 + 冠军/挑战者 | 已有标注与特征快照 | 最高，形成数据飞轮 |
| P1 | 异常检测周期重训 + 漂移监控 | 现有 IsolationForest | 高，止住模型陈旧 |
| P2 | Neo4j 图特征融入评分 | 已有图查询 | 高，捕捉结构性洗钱 |
| P3 | 实体解析与名单智能匹配 | NameMatcher | 中高，降漏报误报 |
| P4 | 可解释性/模型档案/公平性 | 监管硬要求 | 合规必需 |
| P5 | 洗钱模式专项与时序模型 | 上述基础 | 中长期纵深 |

**一句话**：当前最该做的不是引入更复杂的模型，而是**把已经在采集却被丢弃的人工复核标注接回训练闭环**——这是把"叫 AI"变成"是 AI"的最短路径。
