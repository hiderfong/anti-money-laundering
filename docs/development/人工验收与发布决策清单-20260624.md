## 6. 数据质量验收标准

| 类型 | 通过标准 |
|------|----------|
| 演示数据 | 客户、产品、案件、报告、责任人名称应尽量贴近真实业务 |
| 测试痕迹 | `E2E`、`mock`、乱码、空客户名等不应出现在主要演示路径 |
| 关联一致性 | 通知、预警、案件、报告中的业务 ID 能互相对应 |
| 关键字段 | 客户名称、产品描述、报告内容、审批人、责任人不应系统性为空 |
| 长整型 ID | 页面展示和接口跳转不应出现 ID 精度丢失 |

如本轮环境仍保留测试种子数据，应在验收记录中标注“当前为演示/验证环境”；正式生产上线前需清理或替换。

## 7. 阻断等级

| 等级 | 定义 | 处置 |
|------|------|------|
| P0 阻断 | 登录失败、核心接口 500、数据库异常、权限越权、数据不可追溯、无法回滚 | No-Go，必须修复并重新发布候选 |
| P1 重大 | 核心流程可完成但关键字段缺失、图表大面积不可用、重要页面明显错位 | 默认 No-Go；如业务负责人接受，可 Go With Conditions |
| P2 一般 | 个别文案、样式、低频字段、非核心提示问题 | 可记录为后续修复，不阻断 RC |
| P3 优化 | 体验增强、性能优化、布局微调 | 进入后续 backlog |

### 8.1 GO

满足以下条件时可判定 GO：

- 无 P0 阻断项。
- P1 项为 0，或已由决策人明确接受并记录条件。
- 远程服务健康，备份和回滚点可用。
- 业务负责人确认核心演示/验收路径可完成。

后续动作：

1. 创建正式 tag：建议 `v0.9.0`。
2. 生成正式 GitHub Release。
3. 将 prerelease 验证结论关联到正式 release notes。
4. 进入生产运维观察期。

正式 tag 指向建议：

- 若要求“严格指向已部署运行代码”，使用 `f1a7d73c821e09a09879a5f225d99c0dd3c5c9f7`。
- 若要求“正式源码包包含验证文档”，使用最新 `main` 文档提交 `30fc15c4e2275cb416c2e614fa5d9a4f7a4cfe44`；该提交只新增文档，业务代码与远程运行代码一致，且 Release Gate 已全绿。

### 8.2 NO-GO

出现任一 P0 阻断项时判定 NO-GO：

1. 保持 `v0.9.0-rc1` 为 prerelease。
2. 建立修复任务，记录复现步骤、截图、接口响应和日志。
3. 修复后重新触发 GitHub Release Gate。
4. 远程重新部署并生成新的验证报告。
5. 必要时创建 `v0.9.0-rc2`。

### 8.3 GO WITH CONDITIONS

仅允许在无 P0 且 P1 风险可被业务接受时使用。必须记录：

- 接受哪些未解决项。
- 风险边界和影响范围。
- 补救计划与负责人。
- 最晚完成时间。
