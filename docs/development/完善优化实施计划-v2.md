## 4. 阶段 P0-1：验收与测试数据底座

目标：先让“是否变好”可证明。

| 编号 | 工作项 | 涉及范围 | 交付物 | 验收标准 | 预估 |
|------|--------|----------|--------|----------|------|
| T0-1 | 建立 E2E 数据命名规范 | scripts、测试文档 | E2E 前缀规范 | 所有脚本数据可按前缀识别 | 0.5d |
| T0-2 | 增加 E2E 清理脚本 | scripts、DB | `scripts/cleanup-e2e-data.sh` | 连续运行 E2E 不累积脏数据 | 1d |
| T0-3 | 扩展 API E2E 输出 | `scripts/e2e-test.sh` | 失败步骤、请求、响应摘要 | 失败时能定位到具体模块 | 0.5d |
| T0-4 | 引入浏览器级 UI E2E 工具 | frontend | Playwright 或等价方案 | 能执行真实点击/填写/提交 | 1d |
| T0-5 | 建立 E2E 环境说明 | docs/development | 启动、运行、清理说明 | 新机器可按文档跑通 | 0.5d |

建议先做：T0-1、T0-2、T0-3。它们不改变业务行为，但会显著降低后续迭代风险。

## 5. 阶段 P0-2：RBAC 四层一致性

目标：避免“前端隐藏了，但 API 还能调”或“有权限但页面进不去”。

| 编号 | 工作项 | 涉及范围 | 交付物 | 验收标准 | 预估 |
|------|--------|----------|--------|----------|------|
| T1-1 | 固化角色权限矩阵 | docs、DB seed | 角色、菜单、按钮、API 对照表 | 四个角色权限边界明确 | 0.5d |
| T1-2 | 准备多角色测试用户 | migration 或测试脚本 | admin/compliance/investigator/viewer 账号 | E2E 可分别登录各角色 | 0.5d |
| T1-3 | 前端菜单/路由权限 E2E | frontend E2E | 权限 UI 测试 | 无权限菜单不可见，直达受限路由进入 403 | 1d |
| T1-4 | 后端 API 权限校验 | Security、Controller、测试 | 403 集成测试 | 无权限调用写接口返回 403 | 1.5d |
| T1-5 | 按钮权限验收 | Vue 指令、页面 | 创建、编辑、处理等按钮显示测试 | 按钮显示与 `permissions` 完全一致 | 1d |

注意：当前代码已有前端 RBAC 基础，下一步重点是“多角色证据”和“后端接口硬拦截”。

## 12. 推荐立即执行顺序

第一轮建议只做 P0，目标是把项目质量基线立住：

1. T0-1：E2E 数据命名规范。
2. T0-2：E2E 数据清理脚本。
3. T0-3：增强现有 API/前端 smoke 脚本失败诊断。
4. T1-1：角色权限矩阵文档。
5. T1-2：多角色测试用户准备。
6. T2-1：客户建档真实 UI E2E。
7. T2-2：筛查闭环真实 UI/API E2E。
8. T2-3：交易录入与预警触发闭环。

完成第一轮后，项目完善度预计可从约 70% 提升到约 78%-82%，更接近“可持续迭代的准交付版”。

---

## 13. 暂不建议立即做的事项

| 事项 | 暂缓原因 |
|------|----------|
| 大规模视觉重构 | 当前更需要业务闭环和验收稳定性 |
| 引入复杂压测平台 | 先用轻量脚本拿到基线即可 |
| 全量重写 API 封装 | 当前封装可用，先修明确契约不一致 |
| 大规模数据库结构调整 | 状态机和测试闭环未固化前，结构调整风险偏高 |
| 多租户/机构隔离 | 现阶段 PRD 未明确为首批交付目标 |

---

## 14. 下一步确认点

建议下一次执行从以下两个任务开始：

1. **E2E 数据规范与清理脚本**：低风险，能立刻提升测试可重复性。
2. **角色权限矩阵与多角色测试账号**：为后续 RBAC 四层一致性测试做准备。

确认后再进入代码和脚本改造。
