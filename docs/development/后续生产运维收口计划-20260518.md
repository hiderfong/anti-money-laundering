# 后续生产运维收口计划

> 日期：2026-05-18  
> 适用环境：远程生产站点 `https://aml.cnroc.cn` 与本地开发验证环境  
> 说明：本文不记录任何明文密码、token、密钥或服务器登录凭据。

## 1. 当前基线

- 远程后端已由 `systemd` 托管，服务名为 `aml-system`。
- 远程后端以 `prod,no-es` 运行，公网 HTTPS 健康检查已通过。
- MySQL、Redis、Neo4j、MinIO、Kafka 等远程中间件端口已收紧为本机监听。
- Redis AOF 已完成备份与修复，容器已恢复 healthy。
- Elasticsearch 当前未被后端使用，因内存不足反复重启，已停止并取消自动重启。
- GitHub Actions 发布门禁已通过：生产配置检查、密钥扫描、后端测试、前端构建、容器构建、端到端发布门禁。

## 2. 待办优先级

| 优先级 | 事项 | 目标 | 验收标准 |
|--------|------|------|----------|
| P0 | 轮换已暴露过的凭据 | 消除对话和历史操作中暴露凭据带来的长期风险 | GitHub token、服务器 root 密码、项目管理员密码、MySQL/Neo4j/MinIO 管理密码完成轮换，旧凭据不可用 |
| P0 | SSH 登录加固 | 降低服务器被暴力登录或凭据复用攻击的风险 | 新增非 root 运维用户，启用 SSH key 登录，禁用 root 密码登录，保留应急回退路径 |
| P0 | 防火墙收口 | 在端口绑定之外增加网络边界防护 | 仅开放 80、443 和 SSH 管理端口；数据库、中间件和 8080 不可公网访问 |
| P1 | 备份与恢复演练 | 确保生产数据可恢复 | MySQL 定时备份生效，完成一次恢复到临时库的演练并形成报告 |
| P1 | 监控与告警 | 将被动排障转为主动发现 | 监控健康接口、systemd 状态、磁盘、内存、MySQL、证书有效期，并配置告警通知 |
| P1 | 生产中间件编排整理 | 替代当前“开发 compose 栈承载生产中间件”的过渡形态 | 明确常驻服务、按需服务、禁用服务；形成生产 compose 或等价运维脚本 |
| P1 | 远程全量人工验收 | 验证真实生产访问路径下的业务闭环 | 客户、名单、交易、预警、案件、STR、大额报告、通知、图谱等主流程通过并形成验收报告 |
| P2 | Elasticsearch 容量评估 | 为法规库全文检索和审计检索预留能力 | 明确是否启用 ES；如启用，完成内存、heap、磁盘和备份策略评估 |
| P2 | 日志归档与审计留存 | 满足排障和审计追踪要求 | 应用日志、Nginx 日志、关键操作审计日志具备留存周期和归档策略 |
| P2 | 发布自动化完善 | 减少人工操作误差 | 发布脚本覆盖构建、备份、部署、健康检查、回滚提示和验收记录 |

## 3. 推荐执行顺序

1. 轮换所有已暴露过或疑似弱口径的凭据。
2. 新建非 root 运维用户并切换到 SSH key 登录。
3. 配置防火墙，只保留必要公网入口。
4. 建立数据库和关键配置备份，并完成一次恢复演练。
5. 接入基础监控和告警。
6. 整理生产中间件编排，决定 Redis、Neo4j、Kafka、MinIO、Elasticsearch 的常驻策略。
7. 执行远程全量人工业务验收。
8. 再进入新增业务能力、性能优化和可视化增强迭代。

## 4. 本机资源回收策略

本机资源回收只处理本项目开发验证资源：

- 停止本项目本机后端 Java/Maven 进程。
- 停止本项目本机前端 Vite/NPM 开发进程。
- 停止本项目本机 Docker 开发中间件容器。
- 不删除 Docker volumes，不删除数据库数据，不删除构建产物。
- 不影响远程服务器服务。

重新启动本机开发环境时，可按需要执行：

```bash
docker compose -f docker/docker-compose-dev.yml up -d mysql redis neo4j minio kafka
mvn spring-boot:run -Dspring-boot.run.profiles=dev,no-es
cd frontend && npm run dev -- --host 127.0.0.1
```
