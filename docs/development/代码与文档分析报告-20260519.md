## 三、主要问题与风险

### P1 — 需优先处理

| 编号 | 问题 | 证据 | 影响 |
|------|------|------|------|
| P1-1 | **`backups/` 未加入 `.gitignore`，且工作区内含敏感物料** | `backups/admin_hash_*.txt`、`aml_system_before_sync_*.sql.gz`、`aml_local_sync.sql.gz`、`pre-hardening-*/aml_system.sql.gz`、多个 `aml-system*.jar` | 任意 `git add .` 会将数据库快照、口令哈希、构建产物提交进库，敏感信息泄露与仓库膨胀风险高 |
| P1-2 | **测试覆盖偏薄** | 29 测试文件 / 344 源文件（≈8% 文件比）；最大的 `RuleEngineService`(1011 行)、`AlertServiceImpl`(979 行)、`CustomerServiceImpl`(972 行) 等核心服务缺少对应单测 | 核心反洗钱规则/告警逻辑回归保护不足 |
| P1-3 | **数据修复型迁移频繁** | V012 `backfill_case_notification_references`、V013 `repair_legacy_customer_and_alert_notifications`，配合近期提交 `fix: repair notification business references` | 表明历史数据/外键引用曾长期不一致，存在未根治的数据契约问题 |

### P2 — 应安排处理

| 编号 | 问题 | 证据 | 影响 |
|------|------|------|------|
| P2-1 | **包命名遗留 `case_`** | `module/case_` 下 23 个文件；`module/case` 目录存在子目录但 0 文件（空壳） | `case` 为保留字采用 `case_` 绕开，命名不一致，空 `case` 目录易误导 |
| P2-2 | **遗留 `service/*` 与 `module/*` 双结构并存** | 同时存在 `service/{kyc,risk,screening,reporting,core}` 与 `module/{kyc,screening,reporting,...}` | 迁移未收尾，新人难判断"哪个是当前实现" |
| P2-3 | **AppleDouble 资源叉文件污染** | `frontend/._dist`、`backups/**/._assets`、`._index.html` 等 macOS `._*` 文件散落 | 跨平台构建/打包噪声，应清理并在 `.gitignore` 加 `._*` |
| P2-4 | **运维依赖栈过重** | 单系统依赖 MySQL+Redis+Kafka+ES+Neo4j+MinIO 六类中间件 | 部署/运维复杂度高（已通过 `application-no-es.yml` / `no-redis.yml` 部分缓解，但仍偏重） |
| P2-5 | **`sql/migration/` 与 `src/main/resources/db/migration/` 并存** | 根 `sql/migration/V001` 与资源目录 V001~V013 重复 | 迁移源不唯一，存在维护与执行路径混淆风险 |

### P3 — 文档层面

- 文档非常丰富（`docs/development/` 下 30+ 份报告：CI 基线、E2E、发布收口、安全整改、脱敏审计等），但**报告类文档以日期堆叠**（多份"-2026MMDD"报告），缺少单一"当前状态"索引页，读者难判断哪份仍有效。
- 已存在 3 份分析报告（`SOURCE_CODE_ANALYSIS_REPORT.md` 2026-05-04、`项目分析报告-20260515.md`、本报告），建议合并出一份"活文档"并归档历史版本。
- `README.md` 目录结构示意与实际不符（README 描述 `service/kyc` 等传统结构，实际已是 `module/*` 垂直切分）。

1. **立即**：在 `.gitignore` 增加 `backups/` 与 `._*`；核查 `backups/` 内是否已有敏感文件被误提交历史（`git log --all -- backups/`）。
2. **近期**：为三大核心服务（RuleEngine、Alert、Customer）补关键路径单测/集成测试，目标核心域行覆盖率而非文件数。
3. **近期**：彻查 notification 业务引用数据契约，确认 V012/V013 后无新增不一致，必要时加 DB 约束防止再次发生。
4. **规划**：制定 `service/*` → `module/*` 收尾迁移计划，删除空 `case` 目录，统一 `case_` 命名（如改为 `casemgmt`）。
5. **规划**：统一迁移脚本唯一来源（保留 `src/main/resources/db/migration/`，移除根 `sql/migration/` 冗余或在 README 说明用途）。
6. **文档治理**：建立 `docs/development/README.md` 索引，标注每份报告"现行/已归档"，更新根 README 目录结构。
