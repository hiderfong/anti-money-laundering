## 3. 处理过程中的问题

- Flyway 校验曾因已上线迁移文件注释变更而失败。已恢复 `V002`、`V004` 迁移文件原始内容，并提交 `8ed94e5`，避免既有数据库 checksum 不一致。
- Redis 容器存在 AOF 文件损坏并进入重启循环。已先备份数据卷，再用 `redis-check-aof --fix` 修复 manifest，容器已恢复 healthy。当前后端仍按生产配置禁用 Redis，因此该修复不改变业务运行模式。
- Elasticsearch 开发容器出现退出码 `137` 并反复重启，符合内存不足场景。由于当前后端以 `prod,no-es` 运行，已停止该容器并取消自动重启。后续如需启用全文检索，应先独立评估服务器内存与 ES heap。
- 远程中间件仍是开发 compose 栈承载，但对公网暴露面已收敛为本机监听。后续建议迁移到正式 `docker/docker-compose.yml` 或标准生产编排。

## 4. 后续建议

1. 轮换 MySQL root、Neo4j、MinIO 等基础设施 root/admin 类凭据，并保存在服务器本地密钥管理方案中。
2. 为服务器增加防火墙白名单策略，仅开放 80、443、SSH 管理端口。
3. 将远程中间件从开发 compose 栈迁移到正式生产编排，明确哪些服务应常驻、哪些服务按需启用。
4. 如需重新启用 Redis/Elasticsearch，应单独执行容量评估、配置加固和业务回归测试。
5. 将远程部署流程继续沉淀为 `prod` profile 的自动化发布脚本，避免再次回退到 `dev` profile。
