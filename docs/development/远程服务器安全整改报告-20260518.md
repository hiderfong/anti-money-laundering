## 3. 处理过程中的问题

- Flyway 校验曾因已上线迁移文件注释变更而失败。已恢复 `V002`、`V004` 迁移文件原始内容，并提交 `8ed94e5`，避免既有数据库 checksum 不一致。
- Redis 容器存在 AOF 文件损坏并进入重启循环。当前后端已按生产配置禁用 Redis，Redis 容器已停止，不影响业务接口。后续如需重新启用 Redis，应先备份并修复或重建 Redis AOF/volume，再开启 Redis 自动配置。
- 远程中间件仍是开发 compose 栈承载，但对公网暴露面已收敛为本机监听。后续建议迁移到正式 `docker/docker-compose.yml` 或 systemd 管理的标准生产部署。

## 4. 后续建议

1. 为后端 Java 进程增加 systemd 服务，避免手工 `nohup` 方式运行。
2. 规划 Redis 修复窗口，修复 AOF 后再评估是否重新启用 Redis。
3. 轮换 MySQL root、Neo4j、MinIO 等基础设施 root/admin 类凭据，并保存在服务器本地密钥管理方案中。
4. 为服务器增加防火墙白名单策略，仅开放 80、443、SSH 管理端口。
5. 将远程部署流程沉淀为 `prod` profile 的自动化脚本，避免再次回退到 `dev` profile。
