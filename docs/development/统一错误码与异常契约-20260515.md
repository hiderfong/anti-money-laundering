# 统一错误码与异常契约

> 日期：2026-05-15
> 目的：沉淀前后端错误码、异常处理和页面提示映射，避免前端依赖零散字符串判断。

## 一、统一响应格式

所有后端接口默认返回：

```json
{
  "code": 200,
  "message": "success",
  "data": {},
  "timestamp": 1778790000000
}
```

字段说明：

| 字段 | 类型 | 说明 |
|------|------|------|
| `code` | number | 业务错误码或通用 HTTP 语义码 |
| `message` | string | 可直接展示或映射到前端文案的错误消息 |
| `data` | any | 成功时的业务数据；失败时通常为空 |
| `timestamp` | number | 服务端响应时间戳 |

## 二、通用状态码

| 枚举 | code | message | HTTP 语义 | 前端处理建议 |
|------|------|---------|-----------|--------------|
| `SUCCESS` | 200 | `success` | 成功 | 正常处理 `data` |
| `BAD_REQUEST` | 400 | `bad request` | 参数错误 | 表单或顶部错误提示 |
| `UNAUTHORIZED` | 401 | `unauthorized` | 未认证 | 尝试 refresh；失败后跳登录 |
| `FORBIDDEN` | 403 | `forbidden` | 无权限 | 跳 403 或展示无权限提示 |
| `NOT_FOUND` | 404 | `not found` | 资源不存在 | 展示空状态或资源不存在 |
| `TOO_MANY_REQUESTS` | 429 | `too many requests` | 限流 | 读取 `Retry-After`，提示稍后再试 |
| `INTERNAL_ERROR` | 500 | `internal server error` | 服务异常 | 展示通用失败文案并保留 traceId |

## 三、业务错误码段

| 模块 | code 范围 | 已定义错误 |
|------|-----------|------------|
| 用户/认证 | `100xxx` | 用户不存在、用户名重复、密码错误、账户锁定/禁用、Token 过期/非法 |
| 客户/KYC | `110xxx` | 客户不存在、证件号重复、KYC 未完成、评级失败 |
| 名单筛查 | `120xxx` | 名单导入错误、筛查超时、白名单过期 |
| 规则引擎 | `130xxx` | 规则配置错误、规则不存在、规则执行失败 |
| 预警 | `140xxx` | 预警状态错误、预警不存在 |
| 案件 | `150xxx` | 案件已关闭、案件不存在 |
| 监管报送 | `160xxx` | XML 校验失败、报送失败 |

## 四、异常处理映射

| 异常类型 | 返回 code | HTTP 状态 | 说明 |
|----------|-----------|-----------|------|
| `BusinessException` | 异常内置 code | 200 或框架默认 | 业务主动抛出的可预期失败 |
| `RateLimitException` | 429 | 429 | 返回 `Retry-After` 头 |
| `MethodArgumentNotValidException` | 400 | 400 | JSON 请求体参数校验失败 |
| `BindException` | 400 | 400 | 查询/表单参数绑定失败 |
| `TypeMismatchException` | 400 | 400 | 参数类型不匹配 |
| `HttpRequestMethodNotSupportedException` | 400 | 405 | HTTP 方法不支持 |
| `HttpMessageNotReadableException` | 400 | 400 | 请求体无法解析 |
| `AccessDeniedException` | 403 | 403 | 已登录但权限不足 |
| `AuthenticationException` | 401 | 401 | 未认证或认证失败 |
| `Exception` | 500 | 500 | 未知异常兜底 |

## 五、前端处理规则

1. `401`：优先走 refreshToken；refresh 失败后清空登录态并跳转 `/login`。
2. `403`：跳转 `/403` 或在按钮/页面内展示无权限状态。
3. `429`：展示“操作过于频繁”，如响应头含 `Retry-After`，展示等待秒数。
4. `400`：优先展示后端 `message`，用于表单校验。
5. `5xx`：展示通用错误，并在有 `X-Request-Id` 时提示用户保留该编号。

## 六、后续约束

- 新增业务异常时优先扩展 `ResultCode`，不要直接散落 magic number。
- 前端不要硬编码英文 `message` 做业务分支，应基于 `code` 分支。
- 对外部监管、文件导入、批处理等异步场景，应复用 `code + traceId + taskId` 的错误定位方式。
